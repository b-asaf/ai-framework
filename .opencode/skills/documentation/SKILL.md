# Documentation Policy

## When to update docs

After any task, check if the change introduced any of the following:

| Change type | Update required |
|---|---|
| New user-facing flow | Yes |
| New or modified API endpoint or schema | Yes |
| New architectural layer or component | Yes |
| New 3rd party integration | Yes |
| Changed system behavior (startup, config, profiles) | Yes |
| New environment variable or configuration key | Yes |
| Bug fix that changes no behavior | No |
| Test additions or coverage improvements | No |
| Refactor preserving existing architecture | No |
| Style, lint, or formatting changes | No |

## Where docs live

Check `project-overview` for the documentation location. If none is defined:
- Default: `docs/` folder at the repo root
- If `docs/` does not exist, create it

Primary file: `docs/architecture.md`

## docs/architecture.md structure

If creating for the first time:

```markdown
# [Project Name] Architecture

## Overview
[Brief system description]

## System structure
[Repos, how they relate, deployment model]

## Backend architecture
[Layer diagram, key components, data flow]

## Frontend architecture
[Component structure, state, API client]

## API layer
[Endpoints or schema overview]

## Data model
[Key entities and relationships]

## Flows
[One subsection per significant flow, added as they are implemented]

## 3rd party integrations
[External services, SDKs, auth providers]

## Development setup
[How to run locally, required env vars]

## CI/CD
[Pipeline overview, what triggers what]
```

## Architecture diagrams (JSON + SVG)

For projects that need a maintained visual architecture overview, use a JSON-first workflow producing a human-readable SVG. This is more durable than Mermaid or hand-drawn diagrams because the JSON is the source of truth — the SVG is always derived from it.

### Run detection

**First run** (no architecture data exists yet):
1. Read the codebase: entry points, main modules, API surfaces, persistence, external services, existing docs
2. Infer the architecture model from evidence — never guess or invent
3. Write `docs/architecture.json`
4. Generate `docs/architecture.svg` from the JSON

**Refresh run** (architecture data already exists):
1. Read `docs/architecture.json` first — before touching any code
2. Read the current `docs/architecture.svg` to understand its visual contract
3. Read code only to verify drift, additions, removals, and gaps
4. Update JSON to reflect what changed
5. Regenerate or edit SVG from JSON — preserve the existing visual style exactly

On refresh runs, the SVG is not just an output artifact — it is also a layout and style contract. Never overwrite manually curated coordinates, connector routing, or grouping with generic auto-generated layouts.

### What to capture in the JSON

Capture only architecture-relevant facts:
- Primary runtime entry points
- Core feature areas or domains
- Service boundaries
- Major dependencies between features or domains
- Datastores, caches, schedulers, and external systems
- Known uncertainty and evidence gaps

Do **not** capture: raw package trees, every class or interface, generic framework plumbing, or auto-generated dependency graphs.

### Uncertainty policy

When evidence is partial or conflicting, mark it explicitly — never guess:
- `"status": "observed"` — confirmed from code
- `"status": "inferred"` — likely but not directly read
- `"status": "unknown"` — missing evidence; say so

Bad behaviour: guessing ownership from package names, inferring runtime calls from import graphs alone, drawing clean dependencies where the code shows ambiguity.

### Core rules

- Update JSON before touching SVG — always
- If the SVG suggests a change not yet in JSON, fix the JSON first
- Mark all uncertainty explicitly rather than hiding it to make the diagram cleaner
- Preserve unrelated coordinates and connector routing when editing a single feature panel
- Summarise behaviour and dependencies; do not dump package trees

## Writing quality

Every piece of prose generated by any agent — documentation, commit messages, error messages, READMEs, code comments, PR descriptions — must meet these standards. Based on Strunk's *The Elements of Style*.

### The five rules violated most often

**Use the active voice.** Active is more direct and vigorous than passive.
- Weak: *The tests were run by the CI pipeline.*
- Strong: *The CI pipeline ran the tests.*

**Put statements in positive form.** Make definite assertions. Avoid tame, hedging, non-committal language.
- Weak: *He did not think the approach was very useful.*
- Strong: *He thought the approach useless.*

**Use definite, specific, concrete language.** Prefer the specific to the general, the concrete to the abstract.
- Weak: *A period of unfavorable weather set in.*
- Strong: *It rained every day for a week.*

**Omit needless words.** Every word must earn its place.
- *the question as to whether* → *whether*
- *owing to the fact that* → *because*
- *at this point in time* → *now*
- *in order to* → *to*

**Place emphatic words at the end.** The end of a sentence is the position of greatest emphasis. Put what matters most there.

### Self-check before finishing any prose

1. Is every sentence in the active voice unless there is a specific reason for passive?
2. Are there hedging phrases that could be cut or made definite?
3. Are there abstract words where a concrete one would be clearer?
4. Is every word earning its place — or is there filler?
5. Does the most important word or phrase land at the end of its sentence?

- `@architect` — update after every approved HLD; offer architecture JSON + SVG when a new feature changes the system shape
